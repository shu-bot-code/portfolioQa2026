Relatórios de bug

Cadastro aceitando e-mail sem “@” Severidade: Alta
Contexto Formulário principal de cadastro da aplicação.
Passos para reproduzir
  1. Acessar a tela de cadastro.
  2. Preencher nome e senha válidos.
  3. Inserir o e-mail como “usuario.com”.
  4. Clicar em “Registrar”.
Resultado esperado O sistema deve bloquear o envio e informar que o e-mail é inválido.
Resultado atual O sistema aceita e processa o cadastro com e-mail em formato incorreto.
HUD de vida não atualiza após receber dano Severidade: Média
Contexto Gameplay padrão durante a fase 2 do jogo.
Passos para reproduzir
  1. Avançar até a metade da fase 2.
  2. Receber dano no obstáculo “Espinho”.
  3. Observar imediatamente o HUD de vida.
Resultado esperado A interface deve remover instantaneamente 1 ponto de vida do HUD.
Resultado atual O personagem recebe dano, mas o HUD permanece visualmente inalterado.
Checkout retorna erro 500 com carrinho vazio Severidade: Alta
Contexto API backend simulando um e-commerce (rota /api/checkout).
Passos para reproduzir
  1. Autenticar na aplicação e obter o Bearer Token.
  2. Esvaziar todos os itens do carrinho de compras.
  3. Enviar requisição POST para a rota /api/checkout simulando finalização.
Resultado esperado A API deve retornar HTTP 400 (Bad Request) informando que o carrinho está vazio.
Resultado atual Ocorre um HTTP 500 (Internal Server Error) devido à falha ao ler um array null.
Texto sobrepondo botão em resoluções pequenas Severidade: Baixa
Contexto Interface do simulador de delivery em dispositivos menores (Ex: iPhone SE, 320px).
Passos para reproduzir
  1. Abrir o app na visualização mobile com largura de 320px.
  2. Adicionar itens ao pedido.
  3. Avançar até a tela de confirmação de pagamento longo.
Resultado esperado Os botões e o texto de valor devem reorganizar-se (wrap) ou diminuir a fonte para caber.
Resultado atual O valor quebra a linha sem margens, encobrindo metade da área de clique do botão "Confirmar".
